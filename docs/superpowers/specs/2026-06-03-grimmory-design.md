# Grimmory trial deployment on understairs

Status: draft for review
Author: Marco (with Claude)
Date: 2026-06-03

## Context and motivation

The understairs cluster already runs `calibre-web` (linuxserver image, 0.6.26) against a Synology-backed Calibre library. Calibre-web is fine as a reader UI but its ingest story is anemic — there is no drop-folder, no auto-conversion, no metadata enrichment from the web UI.

`calibre-web-automated` (CWA) closes that gap but is a single-maintainer fork with a large open-issue backlog. The other contender, **Grimmory**, is a community fork of BookLore created on 2026-03-12 after BookLore's maintainer lost the community over an AI-PR / undisclosed-CVE / license-pivot incident. Grimmory is younger code but architecturally more deliberate: Spring Boot + MariaDB, supports ebooks/comics/audiobooks, has BookDrop ingest, OPDS, Kobo/KOReader sync.

This spec stands up Grimmory **in parallel** with calibre-web for evaluation. Calibre-web stays in place, untouched. Grimmory gets its own NFS share (cloned from the live library by the user before rollout), its own subdomain, its own everything. If Grimmory wins the bake-off, a follow-up spec decommissions calibre-web. If it doesn't, teardown is `kubectl delete ns grimmory` + remove one line from `clusters/understairs/kustomization.yaml`.

## Goals

1. Run Grimmory on understairs, reachable at `library.homelab.blacksd.tech` from LAN and Tailscale, with the same `SecurityPolicy` allowlist used by other apps.
2. Back Grimmory with a dedicated MariaDB instance running as a sidecar Deployment in the same namespace. No new operator install.
3. Use a fresh NFS share at `/volume1/Apps/grimmory` for the library tree. The user pre-populates it by cloning their local Calibre library before rollout.
4. Local admin account only. Defer OIDC/Zitadel integration until Grimmory survives the trial.
5. Follow the existing native-Kustomize app pattern (`apps/<app>/{base,overlays/understairs}` + `clusters/understairs/<app>.yaml`).

## Non-goals

- Decommissioning `calibre-web`. Out of scope for this spec.
- Migrating live data from calibre-web's library into Grimmory's. The user clones their local library to seed the new share.
- OIDC integration with Zitadel. Deferred — see "Out-of-scope follow-ups".
- High availability for MariaDB. Single replica, `Recreate` strategy.
- Backups for MariaDB beyond the openebs-rawfile PVC reclaim policy. The library tree on NAS is the source of truth; the database is rebuildable from a fresh scan.
- A second deployment to `freeloader`. Scope is understairs only.
- Replacing `csi-driver-nfs` plumbing. Already present after the calibre-web/audiobookshelf rollout — see [2026-06-02-calibre-audiobookshelf-design.md](2026-06-02-calibre-audiobookshelf-design.md).

## Architecture

### Components

Single namespace `grimmory` holds:

- **`grimmory` Deployment** — the Spring Boot app. Mounts `/data` (local PVC, openebs-rawfile) for app state and `/books` (NFS PVC) for the library tree. Reads DB credentials from a generated Secret via env vars.
- **`mariadb` Deployment** — single-replica MariaDB. Mounts `/var/lib/mysql` (local PVC, openebs-rawfile). Root password and Grimmory user password generated by ExternalSecrets `Password` generators.
- **`grimmory` Service** — `ClusterIP` on port 80 → container port 6060.
- **`mariadb` Service** — `ClusterIP` on port 3306.
- **`HTTPRoute`** — `library.homelab.blacksd.tech` → grimmory svc, attached to the shared `external` Gateway in `envoy-gateway-system`.
- **`SecurityPolicy`** — LAN (`192.168.20.0/24`) + Tailscale (`100.64.0.0/10`) allowlist.
- **NFS PV/PVC** — static PV pointing at `/volume1/Apps/grimmory` on `nas.homelab.blacksd.tech`, RWX, no StorageClass (matches the calibre-web pattern).

### Storage model

| Volume                  | Type              | Size  | Purpose                                |
|-------------------------|-------------------|-------|----------------------------------------|
| `pvc-grimmory-data`     | openebs-rawfile   | 2 GiB | Grimmory `/data` — covers, indexes, app state |
| `pvc-grimmory-mariadb`  | openebs-rawfile   | 2 GiB | MariaDB datadir                        |
| `pvc-grimmory-library`  | NFS (static PV)   | 2 GiB | `/volume1/Apps/grimmory` library tree  |

Grimmory's MariaDB-backed metadata model is the reason its DB does **not** go on NFS — unlike calibre-web's `metadata.db`, this is a real RDBMS with real concurrency guarantees and a real datadir. It lives on local block storage where it belongs.

The `pvc-grimmory-library` PV uses the same NFS mount options as the calibre-web library PV (`nfsvers=4.0`, `hard`, `nconnect=4`, `rsize=1048576`, `wsize=1048576`). NFSv4.0 specifically, not 4.1 — see commit `9fd2895` for the reason (4.1 broke on the Synology DSM in use).

### DISK_TYPE=NETWORK

Grimmory exposes a `DISK_TYPE` env var that toggles between `LOCAL` (full UI control over files) and `NETWORK` (delete/move/rename disabled in the UI to avoid clobbering shared mounts). The library is NFS-backed, so we set `DISK_TYPE=NETWORK`. Tradeoff accepted: UI-driven library file management is reduced; BookDrop ingest, metadata edits, reading, sync, OPDS all still work. The user can manage files directly via SMB on the NAS if needed.

### Identity / file ownership

The library NFS share follows the exact same convention already established for calibre-web and audiobookshelf:

- DSM user `k8s-nfs` (UID 1027, GID 100 = `users`), no DSM login rights.
- Pre-chown on the NAS: `chown -R k8s-nfs:users /volume1/Apps/grimmory && chmod -R g+rwsX /volume1/Apps/grimmory`.
- NFS export: "No mapping" squash, R/W, async, allow non-privileged ports, allow subfolder mounts, source IP `192.168.20.0/24`.
- Pod-side `securityContext`: `runAsUser: 1027`, `runAsGroup: 100`, `fsGroup: 100`.

The Grimmory container image (`ghcr.io/grimmory-tools/grimmory`) runs as a non-root user by default; the pinned `runAsUser: 1027` overrides whatever the image bakes in so file ownership on the share stays consistent with the calibre-web share. The MariaDB container (`mariadb:11.4`) defaults to UID 999 (mysql); since its datadir is a local PVC with `fsGroup: 100`, that's fine — no UID gymnastics required there. The MariaDB pod does NOT pin `runAsUser: 1027` because the MariaDB image's entrypoint enforces mysql user ownership of `/var/lib/mysql`.

### Secrets

Two passwords are needed: MariaDB root, and the Grimmory app DB user. Both are generated in-cluster via the ExternalSecrets `Password` generator pattern already established for vaultwarden (`apps/vaultwarden/overlays/understairs/admin-token.externalsecret.yaml`).

Two ExternalSecrets:

- `grimmory-mariadb-root` → Secret with key `password`. Consumed by MariaDB's `MYSQL_ROOT_PASSWORD` env.
- `grimmory-db` → Secret with key `password`. Consumed by both:
  - MariaDB's `MYSQL_PASSWORD` env (paired with `MYSQL_USER=grimmory` and `MYSQL_DATABASE=grimmory` set as static env on the MariaDB Deployment — these initialize the user and database on first start).
  - Grimmory's `DATABASE_PASSWORD` env (or whatever the upstream env name is — confirm at implementation time from Grimmory's docker-compose example).

No SOPS file is needed for this app. ExternalSecrets are not encrypted-at-rest in the repo because the generator produces the password in-cluster; only the resulting Secret exists, and it lives only in etcd.

### Networking

Same shape as calibre-web's `httproute.yaml` (`apps/calibre-web/overlays/understairs/httproute.yaml`), substituting hostname and backend service:

```yaml
hostnames: [library.homelab.blacksd.tech]
backendRefs: [{ name: grimmory, port: 80 }]
```

Same `SecurityPolicy` shape with the LAN and Tailscale CIDRs.

## Per-component design

### Grimmory Deployment

**Image:** `ghcr.io/grimmory-tools/grimmory:<digest>` pinned at implementation time.

**Container port:** 6060 (HTTP, Grimmory default).

**Environment** (confirm names against Grimmory's official docker-compose at implementation time — names below match the docs as of 2026-06-03):

- `TZ=Europe/Rome`
- `DISK_TYPE=NETWORK`
- `DATABASE_URL=jdbc:mariadb://mariadb:3306/grimmory`
- `DATABASE_USERNAME=grimmory`
- `DATABASE_PASSWORD` from Secret `grimmory-db` key `password`
- `BOOKS_PATH=/books` (default, set explicitly for clarity)
- `BOOKDROP_PATH=/books/bookdrop` (a subdir of the library so SMB users can drop files there from the NAS side). **Verify at implementation time that BookDrop's automated file moves are not blocked by `DISK_TYPE=NETWORK`** — the NETWORK flag disables *UI-driven* file ops, but if it also disables BookDrop's internal move-after-import step, BookDrop is broken on NFS and we need an alternative (e.g., point `BOOKDROP_PATH` at a subpath of the local `/data` PVC instead, accepting that drops can't be made via SMB then).

**Volumes:**

- `/data` → `pvc-grimmory-data` (openebs-rawfile, 2 GiB)
- `/books` → `pvc-grimmory-library` (NFS, 2 GiB)

**Resources:**

```yaml
requests: { cpu: 100m, memory: 512Mi }
limits:   { cpu: 1000m, memory: 1536Mi }
```

JVM floor is the reason for the higher requests vs calibre-web. The 1.5 GiB ceiling fits Grimmory's documented "10k+ books → 1-2 GiB" guidance.

**Probes:** HTTP GET `/` on port 6060. Startup probe with `failureThreshold: 60` (Spring Boot cold-start on understairs hardware is slow). Confirm Grimmory exposes a `/actuator/health` endpoint at implementation time and use it if available; otherwise `/`.

**Strategy:** `Recreate` (RWO PVCs, single replica).

**SecurityContext:** `runAsUser: 1027`, `runAsGroup: 100`, `fsGroup: 100`.

**initContainer (optional, decide at implementation time):** A `busybox` initContainer that waits for `mariadb:3306` to accept TCP, so Grimmory doesn't crashloop on first start while MariaDB is still initializing the data directory. Cheap insurance.

### MariaDB Deployment

**Image:** `mariadb:11.4-noble@<digest>` pinned at implementation time.

**Container port:** 3306.

**Environment:**

- `MYSQL_ROOT_PASSWORD` from Secret `grimmory-mariadb-root` key `password`
- `MYSQL_DATABASE=grimmory`
- `MYSQL_USER=grimmory`
- `MYSQL_PASSWORD` from Secret `grimmory-db` key `password`
- `MARIADB_AUTO_UPGRADE=1` (cleans up across minor version bumps)

**Volume:**

- `/var/lib/mysql` → `pvc-grimmory-mariadb` (openebs-rawfile, 2 GiB)

**Resources:**

```yaml
requests: { cpu: 50m, memory: 192Mi }
limits:   { cpu: 500m, memory: 512Mi }
```

**Probes:**

- `livenessProbe`: `exec` `mariadb-admin ping -uroot -p$MYSQL_ROOT_PASSWORD`
- `readinessProbe`: same
- `startupProbe`: same, `failureThreshold: 30` for first-run initdb

**Strategy:** `Recreate`. Do not enable `RollingUpdate` — RWO PVC plus initdb sequencing makes that hazardous.

### Service definitions

```yaml
# grimmory svc
ports: [{ name: http, port: 80, targetPort: 6060 }]

# mariadb svc
ports: [{ name: mysql, port: 3306, targetPort: 3306 }]
```

## NFS PV/PVC

Template, mirroring `apps/calibre-web/overlays/understairs/pv-library.yaml`:

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: grimmory-library
spec:
  capacity: { storage: 2Gi }
  accessModes: [ReadWriteMany]
  persistentVolumeReclaimPolicy: Retain
  storageClassName: ""
  mountOptions:
    - nfsvers=4.0
    - hard
    - nconnect=4
    - rsize=1048576
    - wsize=1048576
  csi:
    driver: nfs.csi.k8s.io
    volumeHandle: grimmory-library
    volumeAttributes:
      server: nas.homelab.blacksd.tech
      share: /volume1/Apps/grimmory
```

Matching PVC binds via `volumeName: grimmory-library` and `storageClassName: ""`.

## DSM-side prep (one-time, by user before merge)

- Create shared folder `/volume1/Apps/grimmory`.
- Grant `k8s-nfs` read/write on it.
- NFS Permissions: No mapping squash, R/W, async, non-privileged ports, allow subfolder mounts, source IP `192.168.20.0/24`.
- SSH to NAS. **These commands are required before the pod can write to the share — without them, NFS writes get permission denied.** DSM's SSH shell requires `sudo` for chown on shared folders:
  ```
  sudo chown -R k8s-nfs:users /volume1/Apps/grimmory/
  sudo chmod -R g+rwsX /volume1/Apps/grimmory/
  sudo mkdir -p /volume1/Apps/grimmory/bookdrop
  sudo chown k8s-nfs:users /volume1/Apps/grimmory/bookdrop
  sudo chmod g+rwsX /volume1/Apps/grimmory/bookdrop
  ```
- Clone the local Calibre library contents into `/volume1/Apps/grimmory` (skipping `metadata.db` — Grimmory doesn't use it).

## Inputs (to confirm at implementation time)

- Exact env var names for Grimmory's DB connection (`DATABASE_URL` vs `SPRING_DATASOURCE_URL` vs other) — pulled from Grimmory's official docker-compose.
- Whether Grimmory exposes `/actuator/health` or a similar healthcheck endpoint distinct from `/`.

## Risks

- **Single-maintainer young fork.** Grimmory is ~3 months old as of writing. Tag movement, breaking changes between versions likely. Mitigation: pin by digest, monitor releases manually, accept that an upgrade may require schema-migration thinking. This is the user's stated tradeoff for getting modern features.
- **MariaDB datadir on `openebs-rawfile`.** Single-replica, no cluster-level backup. If the PVC is lost, the DB is lost. Mitigation: library tree on NAS is the source of truth — a wipe means re-scanning the share into a fresh DB. Acceptable for a trial.
- **`DISK_TYPE=NETWORK` UI limitations.** Delete/move/rename disabled in Grimmory's web UI. Files manageable directly via SMB on the NAS. Acceptable, documented.
- **First-start race between Grimmory and MariaDB.** Mitigated by `initContainer` wait-for-DB or by Kubernetes Deployment restart behavior. Crashloop on first start is recoverable, not corrupting.
- **Resource pressure on understairs.** Grimmory (Java) + MariaDB add ~700 MiB request floor. The understairs mini-PCs already host calibre-web, audiobookshelf, paperless-ngx, vaultwarden, minecraft-bedrock. Check `kubectl top nodes` before merging; if tight, the limit ceiling on Grimmory can be trimmed.

## Out-of-scope follow-ups

- **OIDC via Zitadel.** Wire up only if Grimmory survives the trial. Pattern: Zitadel app/client + SOPS-encrypted Secret in this overlay + appropriate env vars (`OIDC_ISSUER_URL`, `OIDC_CLIENT_ID`, `OIDC_CLIENT_SECRET` — confirm against Grimmory docs at the time).
- **calibre-web decommissioning.** If Grimmory wins, a follow-up spec removes `apps/calibre-web/` from the cluster Kustomization, archives the manifests, and possibly retires the `/volume1/Apps/calibre-library` share. Until then, both run side-by-side.
- **MariaDB backups.** A `mariadb-dump` CronJob writing to NFS could be added if Grimmory becomes long-term. Not worth the manifests for a trial.
- **MariaDB operator.** If we end up running more than one MariaDB workload, revisit `mariadb-operator/mariadb-operator`. Today's single instance does not justify operator install.
- **Kobo/KOReader sync configuration.** Grimmory supports it; configuration is per-user via the Grimmory UI and not a manifest concern.
