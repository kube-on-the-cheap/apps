# Homepage Dashboard on understairs

Status: draft for review
Author: Marco (with Claude)
Date: 2026-06-03

## Context and motivation

The understairs cluster now hosts 5 web-facing apps (Audiobookshelf, Calibre-Web, Minecraft Bedrock, Paperless-ngx, Vaultwarden) plus background services (Flux, openebs-rawfile, csi-driver-nfs, envoy-gateway, cert-manager). Marco wants a single landing page that:

1. Acts as a browser home page for self-hosted apps — one bookmark instead of memorizing five hostnames.
2. Surfaces a few live stats (per-app counts where the apps expose them, cluster CPU/memory).
3. Stays in git as configuration-as-code, matching the rest of the cluster's GitOps model.

[Homepage](https://gethomepage.dev) is the chosen tool: actively maintained, first-class Kubernetes support, widgets for most of the apps already deployed, and configuration is plain YAML — fits the repo's style.

This deploys to understairs only. Freeloader can come later if needed; nothing in the design precludes a second overlay.

## Goals

1. Run Homepage on understairs at `home.homelab.blacksd.tech`, reachable from LAN and Tailscale only (same IP allowlist as the other apps).
2. Surface all five existing user-facing apps as cards on a dashboard, grouped sensibly.
3. Pull live data via service widgets where available: Audiobookshelf, Calibre-Web, Paperless-ngx. Plain link card for Vaultwarden (no widget), Minecraft Bedrock (TCP service, no widget).
4. Show Kubernetes cluster + node CPU/memory via the built-in Kubernetes information widget.
5. Show a small set of external bookmarks (Synology DSM, Tailscale admin, Cloudflare dashboard).
6. Show weather, calendar, date+time, all configured from the static config.
7. Keep widget credentials in a SOPS-encrypted Secret, mounted as env vars consumed by Homepage's `{{HOMEPAGE_VAR_*}}` interpolation in services.yaml.
8. Follow the existing native-Kustomize app pattern (`apps/<app>/{base,overlays/understairs}` + `clusters/understairs/<app>.yaml`).

## Non-goals

- Auto-discovery via `gethomepage.dev/*` annotations on HTTPRoutes. The hybrid choice means we start with static services.yaml and may add annotation-driven entries later — but no annotations are added in this spec. They're documented in a "future work" section.
- Single sign-on / authentication for Homepage. The IP allowlist on the HTTPRoute is the auth boundary. Adding OIDC against authentik/zitadel is out of scope.
- Custom CSS / JS or theming beyond Homepage's default themes.
- Multiple replicas + sticky sessions. Single-replica Deployment is sufficient for a homelab.
- Persistent storage. Homepage is stateless given its config is a mounted ConfigMap; cache/sessions are in-memory.
- A second overlay for freeloader.
- Widget for Vaultwarden (none exists in Homepage upstream as of v1.13.x). Vaultwarden gets a plain link card.
- Widget for Minecraft Bedrock (Homepage's `minecraft` widget targets Java edition's `query` protocol; Bedrock uses a different protocol, no widget).
- Backup/replication for Homepage config. The ConfigMap is in git; that's the backup.

## Architecture

### Repo layout

```
apps/homepage/
  base/
    namespace.yaml
    deployment.yaml         # Deployment + Service + ServiceAccount
    rbac.yaml               # ClusterRole + ClusterRoleBinding (for k8s widget)
    kustomization.yaml
  overlays/understairs/
    kustomization.yaml
    httproute.yaml          # HTTPRoute + SecurityPolicy (LAN + Tailscale)
    secrets.sops.yaml       # SOPS-encrypted Secret with widget credentials
    config/                 # mounted into /app/config
      settings.yaml
      services.yaml
      widgets.yaml
      bookmarks.yaml
      kubernetes.yaml
      docker.yaml           # empty (no Docker integration)
      custom.css            # empty
      custom.js             # empty

clusters/understairs/
  homepage.yaml             # new Flux Kustomization (with decryption.provider: sops)
  kustomization.yaml        # + homepage.yaml
```

### Native Kustomize (no Helm)

There is no official Helm chart from gethomepage. Community charts exist (e.g. `jameswynn/homepage`) but they add a config layer between you and the upstream container that needlessly complicates the SOPS flow. The upstream docs themselves walk users through raw manifests. Matches the pattern used by ghostfolio, calibre-web, audiobookshelf in this repo.

### Image

`ghcr.io/gethomepage/homepage:v1.13.1` (current stable as of brainstorm).
`imagePullPolicy: IfNotPresent` (the repo convention; upstream docs say `Always` for tracking `:latest`, which we don't use).

### Config delivery via configMapGenerator

Homepage reads 8 files from `/app/config/`:
`bookmarks.yaml`, `custom.css`, `custom.js`, `docker.yaml`, `kubernetes.yaml`, `services.yaml`, `settings.yaml`, `widgets.yaml`.

All are sourced from a single ConfigMap generated by Kustomize's `configMapGenerator` from files under `apps/homepage/overlays/understairs/config/`. Kustomize hashes the ConfigMap name (`homepage-config-<hash>`) so changes trigger pod rollouts automatically. The Deployment references the ConfigMap by name; `kustomizeconfig.yaml` is **not** needed for Homepage since the Deployment uses standard `volumes[].configMap.name` (nameReference is handled natively for that field).

The Deployment mounts each file as a `subPath` into `/app/config/<file>`.

### Secrets

A SOPS-encrypted Secret `homepage-secrets` in the `homepage` namespace holds:

- `HOMEPAGE_VAR_AUDIOBOOKSHELF_KEY` — API token from Audiobookshelf admin UI
- `HOMEPAGE_VAR_CALIBRE_WEB_USER`
- `HOMEPAGE_VAR_CALIBRE_WEB_PASSWORD`
- `HOMEPAGE_VAR_PAPERLESS_KEY` — API token from Paperless-ngx user settings

Homepage substitutes `{{HOMEPAGE_VAR_*}}` placeholders in `services.yaml` at startup using the container's environment, which is populated from this Secret via `envFrom`. This is the upstream-documented way to keep credentials out of services.yaml.

The Flux Kustomization for homepage sets `decryption.provider: sops` so kustomize-controller decrypts the Secret before applying. Matches the existing ghostfolio pattern in this repo.

### RBAC

The Kubernetes information widget needs cluster-wide read on namespaces, pods, nodes, ingresses, HTTPRoutes, gateways, and metrics. A `ClusterRole` + `ClusterRoleBinding` grant the homepage ServiceAccount read-only access:

- core: `get,list` on `namespaces`, `pods`, `nodes`
- networking.k8s.io: `get,list` on `ingresses`
- gateway.networking.k8s.io: `get,list` on `httproutes`, `gateways`
- metrics.k8s.io: `get,list` on `nodes`, `pods`

The Traefik `ingressroutes.traefik.containo.us` rule from upstream docs is omitted — this cluster uses envoy-gateway, not Traefik.

### Networking

Same pattern as other apps:

- `HTTPRoute` attached to the shared `external` Gateway in `envoy-gateway-system`.
- Hostname: `home.homelab.blacksd.tech`.
- `SecurityPolicy` allowlists `192.168.20.0/24` (LAN) and `100.64.0.0/10` (Tailscale).
- TLS terminated at the Gateway via cert-manager (existing wildcard cert).

### HOMEPAGE_ALLOWED_HOSTS

Homepage requires this env var to accept requests for any hostname other than `localhost`. Set in the Deployment to:

```
home.homelab.blacksd.tech,$(POD_IP):3000
```

The `$(POD_IP):3000` part keeps the K8s probe working (the probe hits the pod IP directly). `POD_IP` is set from `status.podIP` via the downward API.

### Flux wiring

`clusters/understairs/homepage.yaml` (new Flux Kustomization):

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: homepage
  namespace: flux-system
spec:
  interval: 15m0s
  sourceRef:
    kind: GitRepository
    name: apps
  path: ./apps/homepage/overlays/understairs
  prune: true
  wait: true
  timeout: 10m0s
  decryption:
    provider: sops
```

`clusters/understairs/kustomization.yaml` adds `homepage.yaml` to its `resources` list.

The weather widget needs lat/long from the user. The calendar widget is **not** included by default — it requires an iCal URL. If you have one, replace this paragraph with:

```yaml
- calendar:
    firstDayInWeek: monday
    view: agenda
    maxEvents: 5
    integrations:
      - type: ical
        url: <FILL_IN_ICAL_URL>
        name: Personal
```

### bookmarks.yaml

```yaml
- Infrastructure:
    - Synology DSM:
        - abbr: NAS
          href: <FILL_IN_DSM_URL>
    - Tailscale:
        - abbr: TS
          href: https://login.tailscale.com/admin
    - Cloudflare:
        - abbr: CF
          href: https://dash.cloudflare.com
```

Synology URL is left as a placeholder because DSM listens on `:5001` by default and the actual external URL depends on how Marco accesses it (LAN IP vs. Tailscale name vs. external hostname).

### kubernetes.yaml

```yaml
mode: cluster
```

This tells Homepage to use the in-cluster service account (mounted automatically by Kubernetes) and the `kubernetes` API. `cluster` mode reads from the cluster the pod runs in — no kubeconfig file needed.

### docker.yaml

Empty file (just `---`). Required to exist because Homepage tries to read it; absence triggers a warning. No Docker integration on this cluster.

### custom.css / custom.js

Both empty (just a comment header). Required to exist.

## Deployment specifics

**Resources:**

```yaml
requests: { cpu: 50m, memory: 96Mi }
limits:   { cpu: 500m, memory: 256Mi }
```

Homepage is a Next.js app with a small footprint at idle. The limits accommodate occasional spikes during widget refresh cycles.

**Probes:**

```yaml
readinessProbe:
  httpGet: { path: /api/healthcheck, port: http }
  initialDelaySeconds: 5
  periodSeconds: 10
livenessProbe:
  httpGet: { path: /api/healthcheck, port: http }
  initialDelaySeconds: 30
  periodSeconds: 30
```

**Strategy:** `RollingUpdate` (no PVC, so no Recreate constraint).

**Replicas:** 1 (no HA need on a homelab dashboard).

**Container port:** 3000.

**Service:** ClusterIP, port 80 → targetPort 3000.

## Inputs required from user before implementation

1. **Audiobookshelf API key** — generate in Audiobookshelf admin UI (Config → Users → click your user → "API Token").
2. **Calibre-Web username + password** — your existing admin login (or a dedicated read-only user if you prefer).
3. **Paperless-ngx API token** — Paperless UI → user settings → "Auth Token".
4. **Latitude + longitude** for the weather widget. Trento is roughly `46.067, 11.121` if you don't want to be precise.
5. **iCal URL** for the calendar widget (optional — skip if you don't have one).
6. **Synology DSM URL** as you typically access it (`https://nas.homelab.blacksd.tech:5001`? `https://192.168.20.x:5001`?).

These get filled into the config files at implementation time. The four credentials go through SOPS encryption into `secrets.sops.yaml`.

## Risks

- **Widget polling rate-limits the apps.** Homepage polls each service widget every ~10s by default. Audiobookshelf and Paperless are fine with this; Calibre-Web does a database query per poll which is cheap on a small library but worth noting. If poll rate becomes a problem, the widget supports `cache: <minutes>` to throttle.
- **Hardcoding `:v1.13.1`.** Renovate-style bumps will need to manually edit the tag. Less risk than `:latest`, which would silently break on incompatible upstream changes.
- **HOMEPAGE_ALLOWED_HOSTS misconfig.** If we forget the `$(POD_IP):3000` portion, the kubelet probe fails with 403 and the pod CrashLoops. Catchable in the integration verification step.
- **Secret leak on SOPS misconfig.** If `secrets.sops.yaml` is committed unencrypted, the four API tokens leak. The repo's pre-commit hooks include `detect-private-key` and `gitleaks`; the SOPS pattern is exercised already (ghostfolio).
- **In-cluster URLs for widgets bypass the SecurityPolicy.** That's the point — widgets need authenticated access without the LAN/Tailscale IP filter blocking them — but means a pod inside the cluster can reach all the apps directly. This is already true for any namespace; not a regression. Network policies would tighten this but are out of scope (no NetworkPolicy enforcement on understairs yet).

## Out-of-scope follow-ups

- Add `gethomepage.dev/*` annotations to each app's HTTPRoute so we can migrate from static services.yaml to auto-discovery piecemeal (Homepage's hybrid mode allows both).
- OIDC auth in front of Homepage via Authentik (deployed only on freeloader; would require deploying Authentik to understairs too).
- A second overlay for freeloader, listing freeloader-only apps.
- Custom theme/CSS once we know what we don't like about the default.
- Backup of the static config — currently covered implicitly by git.
